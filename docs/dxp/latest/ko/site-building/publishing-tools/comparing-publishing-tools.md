# 출판 도구 비교

Liferay는 콘텐츠 생성 및 사이트 구축을 위한 두 가지 게시 도구인 출판물과 준비 도구를 제공합니다. 출판은 뛰어난 협업 경험과 다른 Liferay 기능과의 뛰어난 통합을 제공하므로 거의 모든 사용 사례에서 스테이징보다 권장됩니다. 한 가지 예외는 별도의 서버에서 작업 환경과 프로덕션 환경을 호스팅해야 하는 경우입니다. 이를 위해서는 원격 라이브 스테이징을 사용해야 합니다.

[$LIFERAY_LEARN_YOUTUBE_URL$]=https://www.youtube.com/embed/xPVWyMrTa3k

## 출판

출판물은 사이트와 콘텐츠 편집을 위한 변경 내용 추적 도구입니다. 이를 통해 기여자는 프로덕션 환경의 게시된 사이트에 영향을 주지 않고 편집할 수 있는 **출판** 이라는 작업 환경을 만들 수 있습니다. 각 출판물은 회사 범위에 속하며 여러 사이트 및 자산 라이브러리에 걸쳐 변경 사항을 그룹화할 수 있습니다. 프로젝트와 팀에 필요한 만큼 출판물을 만들 수 있습니다.

기본적으로 발행물에 대한 액세스는 해당 작성자로 제한됩니다. 그러나 작성자는 다른 회원을 초대하고 각 출판물에 특정한 역할을 할당할 수 있습니다. 출판물은 독립적이기 때문에 기여자는 서로에게 영향을 주지 않고 변경 사항을 만들고 게시할 수 있습니다. 발행물에 대한 기여자는 자신의 타임라인에 따라 작업하고 준비가 되면 변경 사항을 게시할 수 있습니다.

![Group changes into publications and publish them when ready.](./comparing-publishing-tools/images/01.png)

게시 프로세스 중에 Liferay는 프로덕션과의 충돌을 확인하고 사용자에게 이를 검토하고 해결하라는 메시지를 표시합니다. 일부 충돌은 자동으로 해결될 수 있지만 다른 충돌은 수동으로 해결해야 합니다. 게시된 후 충돌이 없으면 변경 사항이 다른 게시에 전파됩니다. 게시는 게시된 변경 사항에 대한 자세한 기록을 유지하므로 필요한 경우 변경 사항을 신속하게 검토하고 되돌릴 수 있습니다.

이러한 기능을 함께 사용하면 탁월한 편집 및 공동 작업 경험을 제공할 수 있습니다. 인스턴스에 대한 출판물을 설정하려면 [게시물 활성화](./publications/enabling-publications.md)를 참조하세요.

```{important}
Liferay 7.4 U44/GA44부터 출판물은 [워크플로](../../process-automation/workflow.md)와 완전히 통합되므로 출판물에서 추적하는 엔터티에 대한 워크플로를 활성화할 수 있습니다.
```

## 스테이징

스테이징을 사용하면 참가자는 변경 사항을 **라이브** 환경에 게시하기 전에 공유 **스테이징** 환경에서 작업합니다. 사용자는 사이트 및 자산 라이브러리에 대한 준비를 개별적으로 활성화합니다. 활성화되면 Liferay는 원본 사이트 또는 라이브러리의 별도 복사본을 생성하고 동일한 서버([로컬 라이브](./staging/configuring-local-live-staging.md)) 또는 별도 서버([원격)에서 호스팅합니다. 라이브](./staging/configuring-remote-live-staging.md)).

![Liferay creates a separate copy of the original site or library.](./comparing-publishing-tools/images/02.png)

모든 변경 사항은 사이트 또는 자산 라이브러리 범위에 적용됩니다. 사이트 또는 자산 라이브러리 내에서 준비하려는 정확한 페이지와 애플리케이션 데이터를 결정할 수 있습니다. 명시적으로 준비된 데이터에 대한 변경 사항만 추적됩니다. 준비가 되면 변경 사항을 개별적으로 또는 모두 함께 게시하고 실시간 환경은 들어오는 사용자 트래픽을 처리합니다.

준비보다 게시를 권장하지만 배포 모델에 다중 계층 아키텍처를 사용하거나 작업 및 프로덕션 환경에 대한 보안 요구 사항이 다른 경우 원격 라이브 준비가 필요합니다. 이는 출판물이 현재 원격 옵션을 제공하지 않기 때문입니다. 다른 모든 사용 사례에서는 출판물을 사용하세요.

```{important}
Staging을 꼭 사용해야 한다면 개발 초기에 활성화해야 합니다. 활성화되면 스테이징이 즉시 환경 간에 데이터 복사를 시작합니다. 이로 인해 이미 많은 데이터가 존재하는 경우 오류가 발생할 수 있습니다. 스테이징은 데이터를 작은 블록으로 자주 게시하는 것이 가장 잘 작동합니다. 변경 사항을 점진적으로 게시하면 게시 오류를 방지하는 데 도움이 됩니다.
```

## 기능 비교

| 특징                                      | 출판 | 스테이징 |
| :-------------------------------------- | :- | :--- |
| 동일한 서버에서 작업 환경과 프로덕션 환경을 호스팅하세요.        | ✔  | ✔    |
| 별도의 서버에서 작업 환경과 프로덕션 환경을 호스팅하세요.        | ✘  | ✔    |
| 자산 라이브러리 지원                             | ✔  | ✔    |
| 다양한 프로젝트와 팀을 위한 별도의 작업 환경 생성 및 관리       | ✔  | ✘    |
| 변경 블록에 대한 설명                            | ✔  | ✘    |
| 맞춤형 워크플로 사용                             | ✔  | ✔    |
| 게시하기 전에 변경사항 미리보기                       | ✔  | ✔    |
| 작업 환경과 생산 환경의 차이점 검토                    | ✔  | ✘    |
| 동일한 작업 환경에 여러 사이트 또는 자산 라이브러리의 변경 사항 포함 | ✔  | ✘    |
| 변경 사항 게시 시기 예약                          | ✔  | ✔    |
| 이벤트 게시에 대한 알림/경고를 자동으로 보냅니다.            | ✔  | ✔    |
| 게시된 모든 변경 사항에 대한 자세한 기록을 유지합니다.         | ✔  | ✘    |
| 페이지에 게시된 변경사항 되돌리기                      | ✔  | ✔    |
| 게시된 변경 사항을 콘텐츠로 되돌리기                    | ✔  | ✘    |

| 문서 및 미디어        | ✔  | ✔    |
| 양식              | ✔  | ✔    |
| Knowledge Base  | ✔  | ✔    |
| 게시판             | ✔  | ✔    |
| 조직              | ✔  | ✘    |
| 페이지 템플릿         | ✔  | ✔    |
| 사이트 템플렛          | ✔  | ✘    |
| 사용자(연락처 정보만 해당) | ✔  | ✘    |
